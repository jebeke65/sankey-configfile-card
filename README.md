# Sankey Config File Card

A [Home Assistant](https://www.home-assistant.io/) Lovelace **wrapper card** that loads any card's configuration from an external YAML file.

Despite the name — which reflects the use case it was built for (keeping a [sankey-chart](https://github.com/MindFreeze/ha-sankey-chart) in sync with an auto-generated YAML) — this card is **card-agnostic**. It will wrap any Lovelace card and keep its config in an external file that can be regenerated by an integration, script, or external tool, without manual copy/paste into the dashboard.

## Installation

### HACS (custom repository)

1. HACS → ⋮ → Custom repositories
2. Repository: `https://github.com/jebeke65/sankey-configfile-card`
3. Category: `Dashboard`
4. Install, then refresh the browser

### Manual

1. Download the latest `sankey-configfile-card.js` from the [releases page](https://github.com/jebeke65/sankey-configfile-card/releases) (or build from source — see below)
2. Copy to `<config>/www/community/sankey-configfile-card/sankey-configfile-card.js`
3. Settings → Dashboards → ⋮ → Resources → Add resource
   - URL: `/local/community/sankey-configfile-card/sankey-configfile-card.js`
   - Type: `JavaScript Module`
4. Refresh the browser

## Usage

There are two usage patterns, depending on how much of the card config you want to keep in the dashboard.

### Minimal: external file contains the full card config

```yaml
type: custom:sankey-configfile-card
config_url: /local/power/sankey.yaml
cache_bust: true
```

The file at `config_url` is the entire inner card, including its `type:`:

```yaml
# /local/power/sankey.yaml
type: custom:sankey-chart
layout: horizontal
height: 800
nodes:
  - { id: sensor.solar, section: 0, name: Solar }
  - { id: sensor.house, section: 1, name: House }
links:
  - { source: sensor.solar, target: sensor.house }
```

### Merged: inline base + external overrides

Put stable presentation settings (layout, height, styling) inline under `card:`, and let the external file supply only the dynamic parts (nodes, links, entities…):

```yaml
type: custom:sankey-configfile-card
config_url: /local/power/sankey.yaml
cache_bust: true
card:
  type: custom:sankey-chart
  layout: horizontal
  height: 800
  show_names: true
  # nodes and links come from the external file
```

```yaml
# /local/power/sankey.yaml  (only the volatile parts)
nodes:
  - { id: sensor.solar, section: 0, name: Solar }
  - { id: sensor.house, section: 1, name: House }
links:
  - { source: sensor.solar, target: sensor.house }
```

The external YAML is merged **on top of** `card:` — keys present in both win from the file. `config_url` and `cache_bust` are wrapper-level and never part of the inner card.

## Options

| Name         | Type    | Required | Description                                                                                                                  |
| ------------ | ------- | :------: | ---------------------------------------------------------------------------------------------------------------------------- |
| `type`       | string  |    ✓     | Must be `custom:sankey-configfile-card`                                                                                      |
| `config_url` | string  |    ✓¹    | URL or path (e.g. `/local/...`) to a YAML file with the inner card config (or just the parts that override `card:`)          |
| `cache_bust` | boolean |          | When `true`, appends `?_cb=<timestamp>` to `config_url` to bypass the browser cache. Useful when the file changes frequently |
| `card`       | object  |    ✓¹    | Inline card config. Rendered immediately as a fallback, and used as the base that the external file is merged on top of     |

¹ At least one of `config_url` or `card` is required.

### Fallback while loading

When both `card:` and `config_url:` are set, the inline card renders immediately so the dashboard is never blank during the fetch. When the external YAML arrives, the merged result replaces it.

### Authenticated paths

`/local/...` URLs are served behind Home Assistant's auth. The card uses `hass.fetchWithAuth` so they work without exposing the file publicly. Absolute `http(s)://` URLs also work if the target allows CORS from the HA origin.

### Error handling

On fetch failure (bad path, 404, invalid YAML, loaded config has no `type`), the card displays the error inline. Fix the file — the card will recover on the next render or when the dashboard reloads.

## Building from source

```sh
git clone https://github.com/jebeke65/sankey-configfile-card.git
cd sankey-configfile-card
npm install
npm run build
# output: dist/sankey-configfile-card.js
```

## Why not use `config-template-card`?

[`config-template-card`](https://github.com/iantrich/config-template-card) does **client-side templating** — it evaluates JavaScript expressions inside your inline config. That is the right tool when the card *shape* stays fixed and you only want reactive values.

`sankey-configfile-card` loads **a whole different YAML file**. That is the right tool when the card *shape itself* is generated externally (by a HA integration, a script, or any process that emits YAML), and the number of nodes/links/rows can change arbitrarily between regenerations.
